## 总体评价

这版是**文档收口版**，质量比上一版更稳。它没有继续堆 Gate 2 可行性审计，也没有越界声称已经完成最终燃油优化，而是把三个容易混淆的边界说清楚了：

1. **Gate 2 可行性已通过**；
2. **当前轨迹不是最优轨迹**；
3. **`optimized_hmax_sensitivity.csv` 只支持 Gate 2 可行性阶段，不支持最终燃油最优下的高度敏感性**。

这三个点现在基本统一。

# 一、这版最大的改进：不再有旧口径冲突

`brief.md` 已经把“求解器前诊断数值”改成了更准确的范围：

> 输出建模口径、固定路径预检查、无风可行性 Gate、Gate 2 连续可行性验证数值。

这很重要，因为现在已经不是“求解器前”阶段。项目已经完成：

[
N=31/61/121/241
]

网格收敛、ODE 容差敏感性和连续路径审计，并且 (N=241) 已通过 Gate 2 可行性门槛。

因此这版的阶段描述是准确的。

---

# 二、`h_max` 敏感性的边界终于说清楚了

以前 `optimized_hmax_sensitivity.csv` 这个名字容易让人误会为：

> 最终燃油最优条件下的高度上界敏感性。

现在 `approach.md` 明确写出：

> 该表不等同于最终燃油最优条件下的高度上界敏感性。

并进一步说明：

> 当前 `optimized_hmax_sensitivity.csv` 只支持 Gate 2 可行性结论；最终无风燃油优化完成后，仍需在相同 (h_{\max}) 集合下重新计算最优燃油敏感性。

这是必要修正。否则后续写论文时很容易把“可行性高度敏感性”误写成“最优策略高度敏感性”。

---

# 三、Gate 2 阶段可以停止继续扩展

`q3_review13_audit.md` 已经明确写出：

> 下一步应停止继续扩展 Gate 2 可行性审计，进入真正的无风燃油优化。

这个判断正确。

当前继续做 Gate 2 的边际价值已经很低。原因是：

* (N=241) 已通过连续可行性门槛；
* ODE 容差已检查；
* 连续路径约束已检查；
* 历史 Claim 已修正为“历史阶段结论”；
* 高度敏感性的适用范围也已限定。

现在真正缺的是：

[
\boxed{\min(m_0-m_f)}
]

的最终燃油优化结果，而不是更多可行性审计。

# 六、仍有两个小问题

## 1. `approach.md` 的“初始网格”描述略旧

它仍写：

> 初始网格：先用较少节点，例如 `N=31`，通过后再做 `N=61`、`N=121` 网格加密。

现在 Gate 2 实际通过网格已经是：

[
N=241.
]

建议改成：

```text id="k6qwsp"
初始网格可从 N=31 开始，Gate 2 可行性验证已完成 N=31/61/121/241；最终燃油优化建议按 N=61 -> 121 -> 241 continuation。
```

这不是概念错误，只是历史描述需要再同步。

## 2. 下一阶段测试边界还可以更“可执行”

现在已经列出下一阶段要做：

* 重积分验证；
* 燃油恒等式；
* 目标网格收敛；
* 多初值；
* 时间约束；
* 近零推力比例；
* KKT/最优性诊断。

但建议再补一个明确的验收表，例如：

| 检查项                     |                       建议阈值 |   |        |
| ----------------------- | -------------------------: | - | ------ |
| 重积分速度误差                 | (\le10^{-3}\ \mathrm{m/s}) |   |        |
| 重积分高度误差                 |        (\le0.1\ \mathrm m) |   |        |
| 燃油恒等式残差                 |     (\le0.05\ \mathrm{kg}) |   |        |
| 连续路径约束违反                |           (\le10^{-6}) 无量纲 |   |        |
| (                       |            J_{121}-J_{241} | ) | 预先固定阈值 |
| 多初值目标差                  |                     预先固定阈值 |   |        |
| 近零推力比例                  |                   报告，不必硬阈值 |   |        |
| (t_f/t_{\mathrm{base}}) |                   报告并做约束对照 |   |        |

这样本地 coding agent 做最终优化时不会只输出一个 (m_f)。
